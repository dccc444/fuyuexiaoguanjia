# 一站式赴约小管家 - 下阶段开发清单

## 1. 文档目标

本清单用于指导下一阶段开发，统一以下三类要求：

1. 符合产品开发文档中的 MVP 主闭环要求
2. 符合当前“创建页过重，需要改成模块化按需使用”的新需求
3. 确保改造过程中线上产品仍然可以正常打开和使用

本阶段不再把“完整赴约手册”作为默认主流程，而是调整为：

- 用户按需点击某个模块
- 输入最少必要信息
- 直接获得该模块结果
- 当用户完成多个模块后，再可选汇总为完整赴约手册

---

## 2. 本阶段产品方向

### 2.1 方向结论

产品从：

- 重表单创建页
- 一次性生成完整手册

调整为：

- 模块化赴约助手
- 单模块即时使用
- 多模块结果可汇总成手册

### 2.2 这样调整的原因

当前最大问题不是功能不够，而是创建页输入负担太高。

很多用户只想快速解决某一个问题，例如：

- 这座场馆能不能带灯牌
- 散场后怎么回最稳
- 穿什么更方便又出片
- 票档 / 看台有什么区别
- 要不要提前和朋友会合

如果还要求用户先填写完整长表单，再生成整份手册，会明显增加流失。

### 2.3 新主流程

推荐主流程：

1. 用户从首页点击某个模块
2. 系统要求填写该模块最少必要信息
3. 直接返回该模块结果
4. 系统自动沉淀本次赴约草稿
5. 用户需要时，再汇总成完整赴约手册

---

## 3. 本阶段开发原则

### 3.1 产品原则

1. 模块优先，手册次级
2. 最少输入，快速得到结果
3. 渐进式建档，不强迫一次填完
4. 保留完整手册能力，但降级为汇总输出

### 3.2 技术原则

1. 先做前端组织方式重构，不一次性推翻后端
2. 保留现有 `/create`、`/battle-books/:id`、`/my-trips`、`/money` 的可用性
3. 优先复用已有生成接口和场馆规则接口
4. 先用前端草稿状态 + localStorage，后续再补服务端草稿

### 3.3 发布原则

1. 旧链路不删，先新增新链路
2. 新模块稳定后，再逐步把首页入口切过去
3. 任何时刻都要保证线上可打开、可创建、可查看结果

## 6. 模块定义

### 6.1 基础信息模块

作用：

- 建立本次赴约草稿
- 给其他模块提供最小上下文

字段：

- `sceneType`
- `eventName`
- `targetName`
- `city`
- `venue`
- `eventDate`
- `startTime`

规则：

- 这是唯一强制优先模块
- 如果用户直接进入其他模块但缺少基础信息，则先引导补基础信息

### 6.2 场馆规则模块

输入：

- `sceneType`
- `city`
- `venue`

输出：

- 场馆摘要
- 入口提醒
- 禁带物
- 条件可带物
- 纠错入口

优先级：

- P0

### 6.3 路线与返程模块

输入：

- `departureCity`
- `isCrossCity`
- `travelPreference`
- `notes`

输出：

- 到场建议
- 散场返程建议
- 返程风险等级
- 赶车优先动作

优先级：

- P0

### 6.4 门票与位置模块

输入：

- `hasTicket`
- `ticketArea`
- `sceneType`

输出：

- 票务提醒
- 票档 / 看台差异
- 入场建议
- 散场建议

优先级：

- P0

### 6.5 搭子 / 物料 / 会合模块

输入：

- `meetupPlan`
- `merchPlan`
- `companions`
- `isFirstTime`

输出：

- 会合提醒
- 领物料提醒
- 一个人去的建议

优先级：

- P0

### 6.6 后续模块（P1）

后续继续拆：

- 穿搭与妆造
- 补给与吃喝
- 住宿与落脚

1. 每一步都必须能单独开发、测试、上线
2. 每一步都必须保证旧链路继续可用
3. 每一步都必须有明确的“可落地验收标准”
4. 每一步都要兼顾原开发文档的主闭环要求和当前模块化新需求

4. 新增右侧赴约摘要区
5. 建立前端草稿状态
6. 建立 localStorage 自动保存和恢复
7. 保留旧 `/create` 页

#### 涉及文件

- `yueyue-client/src/App.jsx`
- `yueyue-client/src/pages/HomePage.jsx`
- `yueyue-client/src/pages/CreatePlanPage.jsx`

3. 纠错入口可正常提交

## 14. 最终结论

下阶段开发的核心不是继续加长完整手册，而是把产品重构成：

- 用户按需点模块即可用
- 多个模块结果自然沉淀
- 需要时再汇总成完整赴约手册

这条路线同时满足：

1. 开发文档里的 MVP 主闭环要求
2. 你提出的“不要强制完整创建页”的新需求
3. 线上产品必须持续可打开、可使用、可迭代的要求

一句话总结：

`先做模块化赴约助手，后做完整手册汇总；先保证轻量可用，再逐步做深做全。`
