# 路线返程模块功能优化方案实施

## 1. 目标

将当前“路线与返程模块”从经验型建议升级为：

- 地图路线规划
- 到场时间计算
- AI 返程提醒
- 导航外跳能力

用户输入出发地、目的地、活动时间后，系统应直接给出可执行路线，而不只是泛化建议。

---

## 2. 本次优化要解决的问题

当前版本的问题：

1. 只能生成经验型文字建议
2. 不能真正回答“怎么走、多久到、几点出发”
3. 没有地图路线事实依据
4. 不能直接导航
5. 散场返程建议缺少和真实路线的结合

目标效果：

1. 用户输入出发地后可直接搜地点
2. 系统自动用场馆作为目的地
3. 可计算推荐路线、预计时长、建议出发时间
4. 可展示备选路线
5. 可一键跳转地图导航

---

## 3. 产品定位

路线与返程模块升级为：

**地图路线规划 + AI 返程提醒**

产品逻辑分三层：

1. 地图服务负责提供真实路线和耗时
2. 系统负责结合活动开始时间计算最晚出发时间
3. AI 负责解释风险和给出更贴现场的返程建议

---

## 4. 技术选型

推荐地图服务：

- 高德地图

选择原因：

1. 国内城市、场馆、地铁和 POI 覆盖更适合当前产品
2. H5 场景成熟
3. 路线规划、地点搜索、地理编码能力完整
4. 更适合演唱会、音乐节、球赛场馆定位

建议接入的高德能力：

1. 输入提示 API
2. 地理编码 API
3. 路线规划 API
4. 导航外跳链接能力

---

## 5. 功能分期

## 5.1 P0 去程路线规划

输入：

- 出发地
- 目的地
- 活动日期
- 活动开始时间
- 到场缓冲时间
- 出行方式偏好

输出：

- 推荐路线
- 预计时长
- 建议出发时间
- 备选路线
- 地图跳转按钮

## 5.2 P1 散场返程

输入：

- 散场时间
- 返程目的地
- 返程方式偏好

输出：

- 散场返程路线
- 末班车风险
- 打车排队风险
- 返程备选方案

## 5.3 P2 AI 融合

在地图结果基础上增加：

- 更贴现场的到场提醒
- 赶车风险解释
- 散场行为建议
- 是否建议提前离场

- 最稳
- 最快
- 最省钱

## 7. 后端接口设计

建议新增以下接口：

## 7.1 地点联想

`GET /api/maps/place-suggestions?q=关键词&city=城市`

返回结构：

```json
[
  {
    "name": "苏州站",
    "address": "苏州市姑苏区苏站路",
    "district": "姑苏区",
    "location": {
      "lng": 120.6201,
      "lat": 31.3337
    }
  }
]
```

## 7.2 地点解析

`POST /api/maps/geocode`

请求体：

```json
{
  "keyword": "上海体育场",
  "city": "上海"
}
```

返回结构：

```json
{
  "name": "上海体育场",
  "formattedAddress": "上海市徐汇区天钥桥路666号",
  "location": {
    "lng": 121.436,
    "lat": 31.185
  }
}
```

## 7.3 去程路线规划

`POST /api/maps/route-plan`

请求体：

```json
{
  "origin": {
    "name": "苏州站",
    "location": { "lng": 120.62, "lat": 31.33 }
  },
  "destination": {
    "name": "上海体育场",
    "location": { "lng": 121.44, "lat": 31.18 }
  },
  "eventDate": "2026-06-20",
  "startTime": "19:30",
  "arrivalBufferMinutes": 90,
  "transportModes": ["transit", "taxi", "drive"]
}
```

返回结构：

```json
{
  "recommended": {
    "mode": "transit",
    "title": "地铁优先，最稳妥",
    "durationMinutes": 82,
    "distanceText": "约 46 公里",
    "departureTimeRecommended": "17:38",
    "arrivalTimeEstimated": "19:00",
    "stepsSummary": ["步行 8 分钟到地铁", "换乘 2 次", "出站后步行 6 分钟"],
    "riskFlags": ["晚高峰换乘拥挤"]
  },
  "alternatives": [],
  "meta": {
    "destinationArrivalTarget": "19:00"
  }
}
```

## 10. AI 融合方案

地图结果负责“事实层”，AI 负责“解释层”。

两者关系：

1. 地图先给真实路线
2. 系统先算最晚出发时间
3. AI 再补充贴现场的解释

AI 输出重点：

- 是否需要更早出发
- 哪种路线更适合赶车
- 是否建议散场后不要停留
- 是否受场馆规则或人流影响

AI 不直接替代地图，而是在地图结果之上做现场化解释。

## 11. 开发顺序

## 第一步：做 P0 去程规划

完成内容：

1. 地点联想
2. 目的地自动带入场馆
3. 去程路线规划
4. 建议出发时间计算
5. 打开高德导航

目标：

先把“怎么去”做成真正可执行能力。

## 第二步：补多路线比较

完成内容：

1. 最稳路线
2. 最快路线
3. 最省钱路线
4. 路线标签和风险提示

目标：

让用户不用自己对比地图结果。

## 第三步：补散场返程

完成内容：

1. 散场返程路线
2. 末班车风险
3. 打车排队风险
4. 返程备选方案

目标：

把当前“返程文字建议”升级成真实返程规划。

## 第四步：补 AI 解释层

完成内容：

1. 结合场馆规则
2. 结合返程时间压力
3. 结合赶车 / 散场 / 人流场景
4. 输出更贴现场的文字提醒

目标：

让路线模块既真实，又有产品辨识度。

---

## 12. 验收标准

P0 验收：

1. 用户输入出发地可联想到真实地点
2. 目的地可自动带出当前场馆
3. 页面能展示推荐路线和预计时长
4. 页面能展示建议出发时间
5. 点击按钮可跳转高德导航
6. 没走完整创建页也能单独使用

P1 验收：

1. 可展示多条备选路线
2. 可区分最稳 / 最快 / 最省钱
3. 路线切换不丢失结果

P2 验收：

1. 散场返程路线可用
2. 有赶车和末班车风险提示
3. AI 能对路线结果做贴现场解释

---

## 13. 风险点

1. 需要申请高德 API Key
2. 需要评估免费额度和调用成本
3. 场馆名可能存在歧义，目的地应尽量转成经纬度
4. 散场返程属于高波动场景，P0 不建议一步做到“实时”
5. 首期建议优先做外跳导航，不先做复杂地图嵌入

---

## 14. 实施建议结论

本功能最推荐的落地顺序是：

1. 先做高德地图 P0 去程规划
2. 再做多路线比较
3. 再做散场返程
4. 最后叠加 AI 解释层

一句话总结：

**先把路线与返程模块做成真实路线规划工具，再把 AI 建议叠加上去，而不是继续只靠文字经验判断。**
