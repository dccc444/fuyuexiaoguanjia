# 找搭子社交功能实施清单

## 1. 文档目标

本清单用于指导“找搭子社交功能”的开发实施，目标是：

1. 在现有“搭子 / 物料 / 会合模块”基础上增加真实社交能力
2. 让用户可以发布找搭子需求
3. 让其他用户可以浏览需求并主动联系
4. 控制功能复杂度，先做轻量可用版，不一开始做重社区

本阶段不把“找搭子”做成复杂社交平台，而是先做成：

- 可发布
- 可浏览
- 可联系
- 可举报

的轻量需求广场。

---

## 2. 产品目标

### 2.1 核心问题

当前“搭子 / 物料 / 会合模块”只能生成建议，但还不能真正解决：

- 我想找一个人一起去
- 我想找同场次同场馆的搭子
- 我想找人一起进场 / 散场 / 会合 / 拼车 / 拼房

所以本次功能目标是：

把“建议模块”升级成“真实找人能力”。

### 2.2 产品定位

本功能定位为：

**赴约场景下的轻量找搭子需求广场**

不是论坛，不是重社区，不是聊天平台，而是：

- 用户发布需求
- 用户浏览他人需求
- 用户通过联系方式主动联系

---

## 3. 本阶段原则

### 3.1 产品原则

1. 先解决“找人”问题，不先做“社区运营”
2. 先支持发布和浏览，不先做复杂私信
3. 每条需求围绕具体活动场次展开
4. 联系方式可控，不默认开放过多隐私

### 3.2 技术原则

1. 先做轻量 CRUD 闭环
2. 先复用现有路由和页面风格
3. 优先用数据库持久化，不放 localStorage
4. 第一阶段延续现有 `demo-user` 模式

### 3.3 风控原则

1. 必须有举报入口
2. 必须限制敏感内容
3. 必须加免责声明
4. 不开放票务交易、兼职、广告类内容

## 6. 用户流程

### 6.1 发布流程

1. 用户进入 `/buddy/new`
2. 填写活动和找搭子需求
3. 提交帖子
4. 发布成功后进入 `/buddy/:id`

### 6.2 浏览流程

1. 用户进入 `/buddy`
2. 浏览同城 / 同场次帖子
3. 点击进入帖子详情
4. 查看对方需求和联系方式
5. 主动联系对方

### 6.3 我的管理流程

1. 用户进入 `/my-buddy-posts`
2. 查看自己发布的帖子
3. 可删除 / 下架 / 编辑

## 7. 数据结构设计

## 7.1 主表：`buddy_posts`

建议字段：

- `id`
- `user_id`
- `scene_type`
- `event_name`
- `target_name`
- `city`
- `venue`
- `event_date`
- `start_time`
- `ticket_area`
- `intent_type`
- `intent_tags`
- `content`
- `companions_expected`
- `is_first_time`
- `contact_type`
- `contact_value`
- `status`
- `created_at`
- `updated_at`

### 字段说明

- `scene_type`：演唱会 / 音乐节 / 球赛
- `intent_type`：主需求类型
- `intent_tags`：多选标签 JSON
- `content`：自由描述
- `companions_expected`：希望找几个人
- `contact_type`：微信 / 小红书 / 口令 / 其他
- `contact_value`：联系方式内容
- `status`：`active / hidden / closed`

## 7.2 举报表：`buddy_reports`

建议字段：

- `id`
- `post_id`
- `reason`
- `description`
- `reporter_contact`
- `created_at`

## 8. 发布表单设计

## 8.1 基础活动信息

- 活动类型
- 活动名称
- 艺人 / 球队 / 主目标
- 城市
- 场馆
- 日期
- 开始时间
- 票区 / 看台

## 8.2 找搭子需求信息

- 想找什么类型搭子
- 一起做什么
- 是否要求同区 / 同看台
- 是否第一次去
- 想找几个人
- 对搭子的期待

## 8.3 个人说明

- 简短自我介绍
- 当前计划
- 会不会领物料
- 会不会拼车 / 拼房

## 8.4 联系方式

- 联系方式类型
- 联系方式内容
- 是否公开展示

## 13. 风控与审核要求

本阶段必须做：

1. 举报入口
2. 内容长度限制
3. 联系方式校验
4. 敏感词过滤
5. 免责声明

建议提示文案：

- 请勿发布票务交易、广告、兼职、借贷、骚扰信息
- 请勿公开敏感隐私信息
- 平台仅提供搭子信息展示，请自行注意线下安全

本阶段不建议开放：

- 手机号公开展示
- 二维码图片上传
- 自动交换微信
- 站内私信

## 14. 验收清单

### 功能验收

1. 用户可以发布找搭子需求
2. 用户可以浏览广场帖子
3. 用户可以查看详情并联系对方
4. 用户可以举报帖子
5. 用户可以管理自己的发布

### 稳定性验收

1. 列表页可正常加载
2. 详情页可直接访问
3. 表单提交失败有明确提示
4. 删除 / 举报不会导致页面崩溃

### 产品验收

1. 用户能明确看懂每条帖子的活动信息
2. 用户能快速判断是否匹配
3. 联系方式展示清晰
4. 不会让页面像重社区一样杂乱

## 16. 最终结论

“找搭子”功能最适合先做成：

- 围绕具体活动场次
- 以发布需求为核心
- 以浏览和主动联系为主
- 带基本举报和风控能力

一句话总结：

**先做一个轻量、可发布、可浏览、可联系的找搭子广场，验证用户需求后，再逐步升级成更完整的社交撮合系统。**
