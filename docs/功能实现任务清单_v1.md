# 一站式赴约小管家 - 功能实现任务清单 v1

## 1. 规划原则

本清单严格按 `功能模块.docx` 的产品方向拆分，核心目标不是做成泛旅行规划工具，而是做成：

- 演唱会 / 音乐节赴约助手
- 球赛赴约助手
- 兼顾场馆规则、应援、物料、搭子、返程、现场体验的 AI 小管家

任务拆分遵循 4 个原则：

1. 先做主闭环，再做细分功能
2. 先做高频刚需，再做社交和扩展
3. 先做能提升“追星助手辨识度”的模块，再补通用工具能力
4. 每阶段都明确页面、接口、数据字段、后置项

---

## 2. 总开发顺序

建议整体执行顺序如下：

1. 核心赴约闭环
2. 追星 / 观赛专属输入能力
3. 场馆规则库与现场规则能力
4. 演唱会 / 音乐节专区深挖
5. 球赛专区深挖
6. 我的行程 / 分享 / 留存能力
7. 社交 / 物料 / 搭子能力
8. 记账 / 设置 / 运营后台

---

## 3. 阶段 1：核心赴约闭环

### 3.1 目标

先让用户能够完成：

1. 选择场景
2. 填写赴约信息
3. 生成 AI 赴约手册
4. 查看结果
5. 保存与分享

### 3.2 先做页面

1. 首页
   - 场景选择
   - 模块入口
   - 最近安排

2. 创建赴约计划页
   - 活动基础信息
   - 出发与预算
   - 个人关注点
   - 场馆规则预览

3. 赴约手册结果页
   - 赴约安心度
   - 提前提醒你
   - 今天这样去最顺
   - 出发前收一收
   - 给你的贴心提醒
   - 场馆规则

4. 我的安排页
   - 当前手册列表
   - 历史记录

5. 分享页
   - 手册分享展示

### 3.3 先做接口

1. `POST /api/battle-books/generate`
   - 生成赴约手册

2. `GET /api/battle-books/:id`
   - 获取单个手册详情

3. `GET /api/battle-books`
   - 获取我的手册列表

4. `DELETE /api/battle-books/:id`
   - 删除手册

5. `POST /api/battle-books/:id/share`
   - 创建分享链接

6. `GET /api/shared/:token`
   - 获取分享手册

7. `GET /api/venue-rules`
   - 场馆规则预览 / 列表

### 3.4 核心数据字段

#### 赴约基础输入

- `sceneType`
- `eventName`
- `city`
- `venue`
- `eventDate`
- `startTime`
- `departureCity`
- `companions`
- `budgetRange`
- `travelPreference`
- `ticketArea`
- `isCrossCity`
- `isFirstTime`
- `notes`

#### 生成结果

- `score`
- `risks`
- `timeline`
- `checklist`
- `suggestions`
- `venueRules`

### 3.5 可后置

- 多人协作编辑
- 图片版分享海报
- 登录系统
- 数据持久化数据库替换

## 4. 阶段 2：追星 / 观赛专属输入能力

### 4.1 目标

这一阶段是把产品从“普通规划工具”拉成“赴约小管家 / 追星助手”的关键。

### 4.2 先做页面

1. 创建赴约计划页增强
   - 追谁 / 看谁
   - 应援目标
   - 穿搭 / 妆造关注点
   - 物料 / 周边计划
   - 搭子 / 会合计划

2. 赴约手册结果页增强
   - 这次的小目标卡片
   - 更贴近现场的提醒内容

### 4.3 先做接口

复用现有生成接口，不新增主接口，先扩输入结构：

- `POST /api/battle-books/generate`

### 4.4 新增数据字段

#### 演唱会 / 音乐节 / 球赛共用增强字段

- `targetName`
- `supportGoal`
- `outfitFocus`
- `merchPlan`
- `meetupPlan`

### 4.5 生成逻辑要求

模型输出必须能感知这些字段：

1. 如果有 `supportGoal`
   - 风险和建议里要体现应援目标

2. 如果有 `outfitFocus`
   - 清单和建议里要体现穿搭 / 温度 / 行动便利性

3. 如果有 `merchPlan`
   - 到场建议和时间线里要体现领物料 / 买周边节奏

4. 如果有 `meetupPlan`
   - 到场前建议里要体现会合安排

5. 如果有赶车 / 返程备注
   - 散场后建议优先体现返程风险

### 4.6 可后置

- 粉籍系统
- 应援色体系
- 偶像 / 球队收藏

---

## 5. 阶段 3：场馆规则库与现场规则能力

### 5.1 目标

把“内容真实性”和“现场可执行性”补起来。

### 5.2 先做页面

1. 创建页规则预览
2. 手册页场馆规则模块
3. 后续可加规则来源弹层

### 5.3 先做接口

1. `GET /api/venue-rules`
   - 支持全部场馆列表
   - 支持按城市 / 场馆 / 场景匹配

2. 后续可扩展：
   - `POST /api/venue-rules/feedback`
   - 用户纠错反馈

### 5.4 数据结构

#### 场馆规则

- `venueId`
- `venueName`
- `city`
- `aliases`
- `scenes`
- `summary`
- `entryTips`
- `prohibitedItems`
- `allowedOrConditional`
- `transport.publicTransport`
- `transport.parking`
- `sourceNote`
- `sources[]`
- `lastVerified`

### 5.5 当前优先级

先优先做这三层：

1. 主流场馆覆盖
2. 官方规则来源
3. 规则命中稳定性

### 5.6 后置

- 自动抓取更新
- 后台规则编辑器
- 规则版本对比

## 10. 每阶段建议开发顺序

### 第一批立刻做

1. 创建页字段稳定
2. 生成结果把追星字段吃进去
3. 手册页展示“这次的小目标”
4. 场馆规则继续稳定

### 第二批紧接着做

1. 穿搭 / 妆造建议
2. 物料 / 会合提醒
3. 散场返程强化
4. 票档 / 看台专属建议

### 第三批再做

1. 周边补给
2. 住宿推荐
3. 搭子 / 拼房 / 拼车

### 第四批再做

1. 球赛信息模块
2. 社交圈层
3. 记账 / 设置 / 后台

---

## 11. 明确哪些先别做

为了避免产品一开始过散，以下能力建议后置：

1. 真实抢票 / 比价
2. 真正的酒店预订
3. 真正的交通票务购买
4. 实时比分数据
5. 大型社交社区
6. 复杂后台系统

这些都很重，而且不是第一阶段决定产品好不好用的核心。

---

## 12. 下一步推荐

如果按这份清单继续执行，建议马上进入下面这个顺序：

1. 固定阶段 1 + 阶段 2 的字段与页面结构
2. 把生成结果继续往“追星助手”内容调优
3. 优先落地：
   - 穿搭 / 妆造建议
   - 物料 / 搭子 / 会合提醒
   - 散场返程强化
   - 票档 / 看台专属建议

这 4 个模块做完以后，产品辨识度会明显比普通出行工具强很多。
