# 一站式赴约小管家 - 第一批开发任务明细 v2

## 1. 本批次目标

这一批不追求功能铺得很广，只做最能拉开“追星助手 / 赴约助手”和普通工具差距的 4 个功能：

1. 追星字段稳定接入生成结果
2. 穿搭 / 妆造建议
3. 物料 / 搭子 / 会合提醒
4. 散场返程 + 票档 / 看台专属建议

这 4 个做完以后，用户会明显感知到：

- 这不是普通行程表
- 它真的懂演唱会 / 音乐节 / 球赛赴约
- 它会替我想到现场最容易翻车和最想完成的事

---

## 2. 本批次执行顺序

建议严格按这个顺序开发：

1. 功能 1：追星字段稳定接入生成结果
2. 功能 2：穿搭 / 妆造建议
3. 功能 3：物料 / 搭子 / 会合提醒
4. 功能 4：散场返程 + 票档 / 看台专属建议

原因：

1. 功能 1 是底座，不先打通，后面字段都只是“填了但没用”
2. 功能 2 和 3 最能体现“追星助手味道”
3. 功能 4 是强刚需，决定这份手册是不是真的有用

---

## 3. 功能 1：追星字段稳定接入生成结果

### 3.1 功能目标

让用户在创建页填写的这些信息，真正进入 AI 生成逻辑并在结果页可见：

- 去见谁 / 看谁
- 这次最想完成什么
- 穿搭 / 妆造关注点
- 物料 / 周边计划
- 搭子 / 会合计划

### 3.2 用户能看到的效果

用户填写：

- 想好好应援
- 想拍照出片
- 想领物料
- 要和朋友会合
- 一个人去但有点紧张

生成结果就要真的体现这些信息，而不是只生成泛泛的交通建议。

### 3.3 需要改的页面

1. 创建赴约计划页
   - 保证字段完整、文案明确

2. 赴约手册结果页
   - 增加“这次的小目标”展示区

### 3.4 需要补的接口

本功能不新增接口，继续使用：

- `POST /api/battle-books/generate`

但要明确接口 payload 字段已经成为正式字段：

- `targetName`
- `supportGoal`
- `outfitFocus`
- `merchPlan`
- `meetupPlan`

### 3.5 需要补的数据字段

#### BattleBook 输入结构

- `targetName: string`
- `supportGoal: string`
- `outfitFocus: string`
- `merchPlan: string`
- `meetupPlan: string`

#### BattleBook 展示结构

建议新增前端展示分组：

- `fanFocus[]`
  - 可由前端根据输入字段组装
  - 不需要后端单独返回

### 3.7 完成标准

## 4. 功能 2：穿搭 / 妆造建议

### 4.1 功能目标

做出专门针对演唱会 / 音乐节 / 球赛赴约的穿搭建议，不只是天气提醒，而是：

- 好不好出片
- 方不方便走路和排队
- 和场馆 / 天气 / 返程冲不冲突

### 4.2 用户能看到的效果

结果页会出现一个新模块，例如：

- 今天怎么穿更舒服
- 怎么兼顾出片和行动
- 雨天 / 夜场 / 夏场怎么处理

### 4.3 需要加的页面模块

#### 结果页新增模块

建议名称：

- `穿搭和妆造提醒`

展示内容：

1. 穿搭建议 2-4 条
2. 妆造 / 出片提醒 2-3 条
3. 特殊天气提醒 1-2 条

### 4.4 需要补的接口

仍优先复用：

- `POST /api/battle-books/generate`

无需单独新接口，先让 AI 结构化返回。

### 4.5 需要补的数据结构

建议在生成 schema 里新增：

- `styleAdvice`
  - `outfit: string[]`
  - `beauty: string[]`
  - `weather: string[]`

### 4.6 需要改的文件

#### 前端

- [BattleBookView.jsx](D:/赴约小管家开发/yueyue-client/src/components/BattleBookView.jsx)
- [home-v3.css](D:/赴约小管家开发/yueyue-client/src/styles/home-v3.css)

#### 后端

- [generator.js](D:/赴约小管家开发/yueyue-server/src/generator.js)

### 4.7 开发细节建议

AI 提示词要考虑这些条件：

1. `sceneType`
2. `outfitFocus`
3. `eventDate`
4. `city`
5. `venueRules`
6. `isCrossCity`
7. `notes`

### 4.8 完成标准

1. 结果页出现独立的穿搭模块
2. 内容不再泛泛说“注意天气”
3. 能体现追现场语境，例如：
   - 穿裙子是否适合排队
   - 内场是否适合大包
   - 夜场妆造如何兼顾补妆和行动

### 4.9 可后置

- 穿搭图片参考
- AI 穿搭图生成
- 应援色联动推荐

## 5. 功能 3：物料 / 搭子 / 会合提醒

### 5.1 功能目标

这是“追星助手味道”最重的功能之一，要让产品懂这些现实需求：

- 去领物料
- 和朋友会合
- 一个人去怕慌
- 想找搭子
- 想互换小卡 / 周边

### 5.2 用户能看到的效果

结果页新增一个更偏现场社交的小模块。

建议名称：

- `物料和会合提醒`

### 5.3 页面内容建议

展示 3 块内容：

1. `会合提醒`
   - 什么时候会合更稳
   - 先安检还是先见面

2. `物料提醒`
   - 想领物料 / 换物时要留出的时间
   - 大件物料是否会影响进场

3. `一个人去也别慌`
   - 给独自赴约用户的节奏建议

### 5.4 需要补的接口

本阶段先不做真实社交，不新增真实匹配接口。

仍复用：

- `POST /api/battle-books/generate`

### 5.5 需要补的数据结构

建议新增：

- `socialAdvice`

  - `merch: string[]`
  - `solo: string[]`

## 6. 功能 4：散场返程 + 票档 / 看台专属建议

### 6.1 功能目标

这是最强刚需之一。

很多用户的真实焦虑不是“怎么去”，而是：

- 散场后会不会堵
- 会不会赶不上高铁 / 飞机
- 内场 / 看台退场速度差很多怎么办
- 球赛看台和演唱会内场是不是要不同提醒

### 6.2 用户能看到的效果

结果页里这两块内容必须变强：

1. `散场返程提醒`
2. `票档 / 看台专属提醒`

### 6.3 页面调整建议

#### 现有模块增强

1. `今天这样去最顺`
   - 时间线里的 `after_event` 阶段增强

2. `给你的贴心提醒`
   - 单独强化返程和座位差异

3. 建议新增小模块
   - `你的票档 / 看台提醒`

### 6.4 需要补的接口

本阶段仍复用：

- `POST /api/battle-books/generate`

### 6.5 需要补的数据结构

建议新增：

- `seatAdvice`
  - `zoneSummary: string`
  - `entryTips: string[]`
  - `exitTips: string[]`

- `returnAdvice`
  - `riskLevel: string`
  - `tips: string[]`
  - `urgentPlan: string[]`

### 6.6 需要改的文件

#### 前端

- [BattleBookView.jsx](D:/赴约小管家开发/yueyue-client/src/components/BattleBookView.jsx)
- [CreatePlanPage.jsx](D:/赴约小管家开发/yueyue-client/src/pages/CreatePlanPage.jsx)
- [home-v3.css](D:/赴约小管家开发/yueyue-client/src/styles/home-v3.css)

#### 后端

- [generator.js](D:/赴约小管家开发/yueyue-server/src/generator.js)
- [venue-rules.js](D:/赴约小管家开发/yueyue-server/src/venue-rules.js)

### 6.7 提示词重点

模型必须优先考虑：

1. `ticketArea`
2. `sceneType`
3. `departureCity`
4. `isCrossCity`
5. `travelPreference`
6. `notes`
7. `venueRules.transport`

### 6.8 完成标准

1. 不同票档 / 看台区有区别化建议
2. 散场提醒不再只是“早点走”
3. 如果用户备注赶高铁 / 飞机，结果里要有更明确返程优先级
4. 演唱会和球赛的动线提醒要有差异

### 6.9 可后置

- 真实地图接入
- 实时交通数据
- 高铁 / 航班联动
