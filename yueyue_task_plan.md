# 一站式赴约小管家 AI 产品开发任务规划

## 1. 规划原则

本任务规划以 `功能模块.docx` 为主参考，结合产品规划文档与需求整合表格进行收敛。产品第一阶段不做“大而全”的活动社区，而是优先验证一个核心价值闭环：

- 用户输入一场具体活动
- 系统生成一份可执行、可保存、可分享的赴约作战书
- 用户在活动前和活动当天愿意反复打开使用

第一阶段产品形态建议维持为 `H5 Web`，后续再迁移小程序。

## 2. 产品目标拆解

### 2.1 核心目标

围绕“演唱会 / 音乐节 / 球赛”三类场景，建设一套以 AI 生成和规则匹配为核心的赴约管家系统，重点解决以下问题：

- 能不能带东西进场
- 几点到最稳
- 怎么买票、怎么补票、怎么判断票务风险
- 住哪里、怎么去、散场怎么回
- 现场要准备什么
- 一个人去怎么办，是否能找到搭子或群

### 2.2 第一阶段成功标准

- 能创建赴约计划并生成作战书
- 能输出场馆规则与防踩坑提醒
- 能给出交通、住宿、入场、散场建议
- 能保存、分享、二次查看
- 至少支持 3 个真实演示场景：演唱会、音乐节、球赛

## 3. 产品范围与版本边界

### 3.1 P0 必做能力

1. AI 创建赴约作战书
2. 场馆规则库
3. 票务助手
4. 交通住宿与散场规划
5. 现场指挥中心
6. 我的行程
7. 基础分享与反馈闭环

### 3.2 P1 增强能力

1. 同场社交
2. 现场生存包
3. 座位视野助手
4. 记账本与 AA 分账

### 3.3 P2 后续能力

1. 轻资讯与赛事数据
2. 拍摄记录助手
3. 妆造 / 穿搭 / 应援色深度内容
4. 海外观赛扩展

### 3.4 首版明确不做

- 小程序正式发布流程
- 自动购票或票务交易闭环
- 酒店机票真实交易闭环
- 重社区、复杂私信系统
- 重资讯内容流
- 高风险的“对骂群”等情绪社区模块

## 4. 任务总拆解

## 4.1 产品与需求层

1. 输出统一 PRD
   - 统一三份材料中的产品定义、模块边界、页面结构、优先级
   - 明确 P0/P1/P2 和不做项
2. 输出用户旅程图
   - 想去 -> 买票 -> 出发 -> 入场 -> 现场 -> 散场 -> 复盘
3. 输出场景模板
   - 演唱会模板
   - 音乐节模板
   - 球赛模板
4. 输出规则字段定义
   - 可带/禁带/不建议带
   - 证件/儿童/实名/寄存/入口/安检/散场
5. 输出验收标准
   - 作战书是否完整
   - 风险提醒是否可执行
   - 分享与保存是否顺畅

## 4.2 AI 能力层

1. 定义 AI 核心输出结构
   - 赴约成功率评分
   - 最大风险 3-5 条
   - 时间线
   - 入场清单
   - 交通住宿建议
   - 散场返程建议
2. 设计 Prompt 与生成链路
   - 用户输入理解
   - 场景模板选择
   - 场馆规则召回
   - 风险打分
   - 作战书生成
3. 建立规则 + 模板混合生成机制
   - 规则库负责“准”
   - AI 负责“个性化整合与表达”
4. 建立可追溯输出
   - 每条规则附来源/更新时间
   - 每条高风险提醒附触发原因
5. 建立纠错反馈闭环
   - 用户反馈规则不准
   - 运营后台修正规则
   - 规则更新后重新生成

## 4.3 数据与内容层

1. 建设首批场馆规则库
   - 北京：鸟巢、华熙、工体
   - 上海：梅奔、虹口、上体
   - 杭州奥体
   - 南京奥体
   - 成都凤凰山
   - 广深 1-2 个热门场馆
2. 设计活动与场馆数据模型
   - 活动
   - 场馆
   - 场景类型
   - 规则项
   - FAQ
   - 作战书
3. 建设票务知识库
   - 开票提醒规则
   - 实名/退票/转让/纸质票/电子票 FAQ
   - 不同场景的票务风险提示
4. 建设交通住宿模板库
   - 省钱方案
   - 省心方案
   - 快速方案
5. 建设现场生存包模板
   - 补水
   - 天气
   - 穿搭
   - 应援物
   - 充电与拍摄

## 4.4 前端产品层

1. 首页
   - 场景选择：演唱会 / 音乐节 / 球赛
   - 快速创建赴约计划入口
2. 创建页
   - 活动名称、城市、场馆、日期、预算、人数、票区、偏好
3. 作战书结果页
   - 成功率评分
   - 最大风险
   - 行程时间线
   - 可带/禁带清单
   - 交通住宿建议
   - 散场返程建议
4. 活动当天指挥中心
   - 倒计时
   - 当前阶段动作提醒
   - 清单勾选
   - 天气/补给/入口提示
5. 我的行程
   - 当前行程
   - 历史作战书
   - 分享记录
6. 设置页
   - 粉籍/主队偏好
   - 预算偏好
   - 通知偏好

## 4.5 后端与平台层

1. 用户体系
   - 游客态创建
   - 微信/手机号登录预留
2. 作战书生成服务
   - 输入校验
   - 模板匹配
   - AI 生成
   - 结果缓存
3. 规则知识库服务
   - 场馆规则维护
   - FAQ 查询
   - 版本更新时间
4. 行程规划服务
   - 时间线组装
   - 风险规则计算
   - 方案推荐
5. 分享与反馈服务
   - 分享链接
   - 反馈纠错
   - 埋点统计
6. 运营后台
   - 场馆规则管理
   - 活动模板管理
   - 用户反馈处理

## 5. 按模块的开发任务清单

### 5.1 模块 A：创建赴约作战书

任务：

1. 定义输入字段与表单结构
2. 设计三类场景的默认模板
3. 建立输入校验与缺省值策略
4. 完成作战书生成接口
5. 完成结果页展示与保存逻辑

交付物：

- 表单字段文档
- 作战书 JSON 结构
- H5 创建流程

### 5.2 模块 B：赴约成功率评分与防踩坑雷达

任务：

1. 定义评分维度
   - 票务确定性
   - 到场时间充足度
   - 入场风险
   - 天气与补给准备度
   - 散场返程可行性
2. 建立基础打分规则
3. 设计 3-5 条高风险输出格式
4. 输出“如何提升分数”的可操作建议

交付物：

- 评分规则文档
- 风险规则清单
- 风险输出模板

### 5.3 模块 C：场馆规则库

任务：

1. 定义规则字段模型
2. 建立首批场馆数据录入模板
3. 建立活动类型差异规则
4. 建立来源与更新时间字段
5. 实现用户按场馆/活动查询

交付物：

- 场馆规则表结构
- 首批场馆规则数据
- 规则查询接口

### 5.4 模块 D：票务助手

任务：

1. 梳理演唱会与球赛的票务问答场景
2. 建立票务 FAQ 与风险文案模板
3. 支持开票提醒订阅
4. 支持票务渠道说明与注意事项
5. 支持实名/退票/转让/纸电票说明

交付物：

- 票务 FAQ 知识库
- 提醒订阅流程
- 票务说明页面

### 5.5 模块 E：交通住宿与散场规划

任务：

1. 设计省钱/省心/快速三类方案
2. 输出到场建议时间
3. 输出散场返程建议
4. 输出酒店位置对比模板
5. 输出接驳/地铁/打车提醒

交付物：

- 时间线生成规则
- 路线与住宿推荐模板
- 散场方案模板

### 5.6 模块 F：现场指挥中心

任务：

1. 设计活动当天页面结构
2. 支持倒计时与阶段提醒
3. 支持清单勾选
4. 支持天气与补给提示
5. 支持入口/寄存/厕所/餐饮提示

交付物：

- 当天指挥中心页面
- 清单组件
- 提醒组件

### 5.7 模块 G：同场社交

任务：

1. 先设计为轻量入口，不直接做复杂社区
2. 支持发帖模板
3. 支持搭子/拼车/拼房/换物标签
4. 支持身份标签与安全提示
5. 支持举报与屏蔽

交付物：

- 社交 MVP 方案
- 安全边界规则
- 发帖与匹配原型

备注：该模块建议在 P0 闭环稳定后进入正式开发。

## 6. 推荐研发排期

### 第 1 周：产品收敛与底座设计

- 完成 PRD 合并版
- 完成信息架构与页面流程图
- 完成数据模型设计
- 完成 AI 输出结构与评分规则初稿

### 第 2 周：规则库与作战书生成 MVP

- 落首批场馆规则数据
- 完成创建表单
- 完成作战书生成接口
- 完成作战书结果页

### 第 3 周：票务 + 行程规划

- 完成票务 FAQ 与提醒订阅
- 完成交通住宿模板
- 完成时间线与散场规划
- 接入成功率评分与防踩坑雷达

### 第 4 周：现场指挥中心 + 我的行程

- 完成当天行动页
- 完成清单勾选与提醒展示
- 完成我的行程与历史记录
- 完成分享链路

### 第 5 周：测试与案例打磨

- 演唱会案例验证
- 音乐节案例验证
- 球赛案例验证
- 修正规则准确率和提示质量

### 第 6 周：灰度验证

- 小范围真实用户试用
- 收集保存率、分享率、反馈率
- 判断是否进入小程序化或扩场景

- 管理 PRD 与验收标准
- 推进真实案例验证

### AI / 后端

- 规则库、知识库、生成服务
- 评分引擎与风险引擎
- 分享、反馈、埋点

### 前端

- H5 页面与交互
- 作战书展示体验
- 活动当天指挥中心

### 运营 / 内容

- 首批场馆规则录入
- FAQ 审核与更新
- 用户反馈修正

## 8. 核心风险与控制

1. 需求过散
   - 控制方式：第一阶段只验证“单场作战书”
2. 场馆规则不准
   - 控制方式：所有规则保留来源和更新时间，先人工审核
3. AI 输出空泛
   - 控制方式：采用规则库 + 模板 + AI 混合生成，不纯靠大模型自由发挥
4. 社交模块风险高
   - 控制方式：先做轻量发帖与匹配，不做重私信社区
5. 票务合规风险
   - 控制方式：首版仅做提醒、FAQ、风险说明，不直接撮合交易

## 9. 最终建议的 MVP 组合

最适合首版上线的组合不是把 `功能模块.docx` 里的所有模块平铺开发，而是收敛为下面 6 个核心能力：

1. 创建赴约计划
2. 场馆规则库
3. 成功率评分 + 防踩坑雷达
4. 交通住宿与散场规划
5. 现场指挥中心
6. 我的行程 + 分享 + 纠错反馈

这样既保留了“一站式赴约小管家”的差异化，也能在较短周期内做出真正可验证的 AI 产品闭环。
