## 一、总原则

这产品不是资讯站，也不是普通旅游规划工具。

真正的产品主轴是：

`把一次线下活动，从“想去”安排成“能顺利到场并玩得舒服”。`

所以开发顺序必须围绕高频执行焦虑来排：

1. 场馆规则
2. 行程规划
3. 票务执行
4. 现场生存与返程
5. 同场社交
6. 轻资讯补充

## 三、演唱会 / 音乐节模块开发方案

## 3.1 产品目标

解决用户这些问题：

- 穿什么、化什么更方便又好看
- 周边吃什么、补给怎么安排
- 住哪里散场更顺
- 跨城交通怎么安排
- 票档、入口、代抢比价怎么处理
- 找搭子、换物料、拼房拼车怎么做

## 3.2 功能拆分

### A. 衣：穿搭 / 妆造

功能点：

- 天气 + 场馆规则联动穿搭建议
- 妆造提醒
- 出片建议
- 必备补妆 / 补给清单

页面落点：

- 创建页偏好选择
- 手册页 `衣` 模块
- 活动当天提醒卡

数据字段：

- `outfitFocus`
- `weatherLevel`
- `photoGoal`
- `comfortPriority`

接口：

- `POST /api/style-advice/generate`
- `GET /api/weather/scene`

### B. 食：周边餐饮 / 美食攻略

功能点：

- 进场前补给建议
- 场馆周边餐饮
- 场内补给提醒
- 散场后夜宵建议

页面落点：

- 创建页偏好配置
- 手册页 `食` 模块
- 场馆页附属推荐

数据字段：

- `foodPlan`
- `mealWindow`
- `crowdTolerance`
- `lateNightPlan`

接口：

- `POST /api/food-advice/generate`
- `GET /api/venues/:id/nearby-food`

### C. 住：住宿推荐 / 拼房

功能点：

- 场馆附近 / 地铁方便 / 性价比 三类住宿策略
- 拼房意向
- 返程前住哪更顺
- 多日活动住宿建议

页面落点：

- 创建页 `住哪里更顺手`
- 手册页 `住` 模块
- 后续社交里的拼房功能

数据字段：

- `stayPlan`
- `stayNights`
- `stayStrategy`
- `shareRoomIntent`

接口：

- `POST /api/stay-advice/generate`
- `GET /api/cities/:city/stay-zones`

### D. 行：公交 / 跨城交通 / 拼车 / 旅游攻略

功能点：

- 出发地到活动城市交通建议
- 到场路线
- 入场前节奏
- 散场返程方案
- 拼车意向
- 周边游轻攻略

页面落点：

- 创建页主搜索与路线配置
- 手册页 `行` 模块
- 我的行程

数据字段：

- `departureCity`
- `travelPreference`
- `arrivalDeadline`
- `returnPressure`
- `isCrossCity`
- `rideShareIntent`

接口：

- `POST /api/travel-advice/generate`
- `GET /api/routes/venue`
- `GET /api/transport/hubs`

### E. 票：代抢比价

功能点：

- 开票提醒
- 补票提醒
- 票档/入口/看台差异
- 代抢价格记录
- 渠道说明与风险提醒

页面落点：

- 创建页票档字段
- 手册页 `票` 模块
- 票务助手子页

数据字段：

- `ticketArea`
- `ticketBudget`
- `ticketNeedCount`
- `ticketSourcePreference`
- `remindOnSale`

接口：

- `POST /api/ticket-reminders`
- `GET /api/ticket-guides/:eventId`
- `POST /api/ticket-price-tracker`

### F. 社交 / 物料：互换社区 / 找搭子 / 粉籍圈

功能点：

- 找同场搭子
- 物料互换
- 粉籍标签
- 拼车拼房
- 会合点提醒

页面落点：

- 创建页偏好配置
- 手册页 `社交` 模块
- 后续专区社区页

数据字段：

- `meetupPlan`
- `fandomTag`
- `merchPlan`
- `socialIntent`
- `groupJoinIntent`

接口：

- `POST /api/social-posts`
- `POST /api/matchmaking`
- `GET /api/events/:id/community`

## 四、球赛模块开发方案

## 4.1 产品目标

球赛不是照搬演唱会，而是要突出：

- 主客场 / 看台 / 分区
- 比赛信息
- 球迷社交
- 现场礼仪
- 海外/跨城看球动线

## 4.2 功能拆分

### A. 行程规划

按文档要求：

- 预算
- 机票 / 高铁 / 火车
- 酒店
- 场外交通
- 夜宵
- 海外场景下的签证或出境准备

页面落点：

- 创建页球赛专属配置
- 球赛手册主模块

数据字段：

- `matchBudget`
- `flightIntent`
- `stayIntent`
- `postMatchPlan`
- `crossBorderNeeded`

接口：

- `POST /api/match/travel-plan`
- `GET /api/match/arrival-options`

### B. 信息模块

按文档要求：

- 比分
- 数据
- 战报
- 赛程

定位：

- 轻资讯
- 不做资讯站
- 主要服务“这场值不值得关注、比赛前后补充看什么”

页面落点：

- 球赛专区信息页
- 手册页附属信息卡

数据字段：

- `teamId`
- `leagueId`
- `matchId`

接口：

- `GET /api/match/:id/score`
- `GET /api/match/:id/stats`
- `GET /api/team/:id/fixtures`

### C. 现场观赛指南

按文档要求：

- 看台
- 热狗 / 餐饮
- 礼仪
- 设施

页面落点：

- 手册页 `现场观赛指南`
- 场馆规则子页

数据字段：

- `seatZone`
- `homeAwaySide`
- `facilityNeeds`

接口：

- `GET /api/stadium-guides/:venueId`
- `POST /api/seat-advice/generate`

### D. 社交与情绪宣泄

按文档要求：

- 球队圈
- 吐槽
- 匹配
- 赛后交流

注意：

- 这个模块需要严格审核
- 不要真的做“对骂群”，产品上应改为情绪讨论区 / 赛后吐槽区

页面落点：

- 球赛社区页
- 赛后话题页

数据字段：

- `teamSupport`
- `socialIntent`
- `watchPartyIntent`

接口：

- `POST /api/match/community/posts`
- `POST /api/match/community/match`

### E. 应援与物料

按文档要求：

- 球衣提示
- 周边代购

页面落点：

- 球赛手册页 `应援与物料`

数据字段：

- `jerseyIntent`
- `merchIntent`

接口：

- `POST /api/match/merch-advice`

## 五、公共模块开发方案

## 5.1 首页（场景选择）

场景：

- 演唱会 / 音乐节
- 球赛

职责：

- 选场景
- 进入创建计划
- 进入历史安排

接口：

- 无强依赖，可静态 + 少量推荐数据

## 5.2 创建赴约计划

这是整产品最关键的 P0 页面。

统一输入：

- 场景
- 活动名称
- 目标艺人/球队
- 城市
- 场馆
- 日期
- 预算
- 同行人数
- 偏好

输出：

- 手册生成请求

核心接口：

- `POST /api/battle-books/generate`
- `GET /api/venue-rules`

## 5.3 场馆规则库

这是所有场景的公共底座。

必须支持：

- 禁带物
- 可带 / 条件可带
- 证件要求
- 儿童规则
- 寄存
- 入口
- 安检强度
- 散场与返程注意事项

核心数据表：

- `venues`
- `venue_rules`
- `venue_rule_sources`
- `venue_match_aliases`

接口：

- `GET /api/venue-rules`
- `GET /api/venue-rules/:venueId`

## 5.4 我的行程

能力：

- 当前安排
- 历史安排
- 再次生成
- 复制计划
- 分享

接口：

- `GET /api/battle-books`
- `GET /api/battle-books/:id`
- `DELETE /api/battle-books/:id`

## 5.5 分享页

能力：

- 可转发摘要
- 打开完整手册

接口：

- `POST /api/battle-books/:id/share`
- `GET /api/shared/:token`

## 5.6 记账本 & AA 分账

能力：

- 交通 / 门票 / 酒店 / 餐饮 / 周边支出记录
- 人均计算
- 分账明细

数据表：

- `expense_books`
- `expense_items`
- `expense_members`

接口：

- `POST /api/expense-books`
- `POST /api/expense-items`
- `GET /api/expense-books/:id`

## 5.7 设置

能力：

- 粉籍 / 主队偏好
- 通知偏好
- 默认预算 / 默认出行风格

数据表：

- `user_preferences`

接口：

- `GET /api/me/preferences`
- `PUT /api/me/preferences`

## 5.8 通知与提醒

能力：

- 开票提醒
- 活动前提醒
- 天气提醒
- 散场返程提醒

接口：

- `POST /api/reminders`
- `GET /api/reminders`

## 十一、最终结论

完整实现时，产品不应该按“页面好看不好看”来排，而要按：

1. 公共底座先立住
2. 演唱会 / 音乐节先做深
3. 球赛模块再跟进
4. 最后补长期留存和社区

最核心的一句话是：

`公共模块决定产品能不能跑，演唱会/音乐节模块决定产品像不像追星助手，球赛模块决定产品能不能扩成真正的一站式赴约平台。`
